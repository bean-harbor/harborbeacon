所有参与者需先阅读:

### 关键背景

**项目愿景**: 构建一个隐私优先、本地智能的混合计算系统，能够：
- 在本地处理简单任务（避免隐私泄露）
- 智能判断何时调用云服务
- 完整脱敏敏感数据后才发送给云端
- 支持多模态（文本、图像、音频、视频）

## 会议议程 (120 分钟)

### 第 1 部分: 架构与设计 (30 分钟)

**Speaker**: 架构师

#### 话题 1: 三层执行路由 (10 分钟)

展示架构图，解释:
- **LOCAL (简单)**: 文本理解、本地文件搜索 → Ollama 本地推理
- **HYBRID (中等)**: 多文件分析、需要部分云增强 → 本地初步 + 云精细
- **CLOUD (复杂)**: 深度分析、实时翻译 → 完全脱敏后云端处理

**讨论问题**:
- Q1: 本地模型 (Mistral 7B / LLaMA 13B) 性能预期与现有系统匹配吗?
- Q2: 三个路由级别的复杂度界线是否合理? (建议: < 30 = LOCAL, 30-70 = HYBRID, > 70 = CLOUD)
- Q3: 是否需要四层或两层?

#### 话题 2: 多模态 RAG 能力 (10 分钟)

展示 RAG 数据流:

```
文本 → sentence-transformers → Milvus
图像 → CLIP → Milvus  
音频 → Whisper → 文本 → Milvus
视频 → 关键帧+Whisper → Milvus
```

**讨论问题**:
- Q4: 是否所有数据都需要向量化? (建议: 是, 除非文件太大)
- Q5: Milvus vs Weaviate vs FAISS? 根据预期数据量?
  - 小规模 (<1M vectors): FAISS or SQLite
  - 中规模 (1M-100M): Milvus
  - 大规模 (>100M): 分片 Milvus
- Q6: 向量维度选择 384 (MiniLM) 还是 768 (更精确但慢)?

#### 话题 3: 智能脱敏流程 (10 分钟)

展示脱敏流程:

```
检测 PII (邮箱/电话/SSN) 
  ↓
生成占位符并加密映射
  ↓
发送脱敏内容到云端
  ↓
云端返回结果
  ↓
本地反脱敏恢复原值
```

**讨论问题**:
- Q7: PII 检测规则完整吗? 需要补充什么?
  - 基础: 邮箱、电话、身份证、银行卡
  - 进阶: 社交媒体 ID、位置信息、家庭成员名字
- Q8: 脱敏粒度: 整个句子 vs 仅 PII 值?
- Q9: 密钥管理策略? (建议: Fernet 对称加密, 存储在 Secret Manager)

#### 话题 3.5: HarborBeacon IM 接入架构 (10 分钟)

展示 HarborBeacon 架构:

```
[IM Channels]          [HarborBeacon]              [Orchestrator]
飞书/企微/Telegram  →  channels.py (路由)     →   planner / router
Discord/钉钉/Slack     mcp_adapter.py (MCP桥)     policy / audit
MQTT                   autonomy.py (自主级别)      executors → HarborOS
```

**关键说明**:
- HarborBeacon 是基于 ZeroClaw 的二次开发，预装在 HarborOS 中（同一台机器）
- 开机后手动填写并校验 IM 通道凭证（飞书/企微/Telegram/Discord/钉钉/Slack/MQTT）
- 自主级别: ReadOnly（只读安全）/ Supervised（需审批）/ Full（完全自主）

**讨论问题**:
- Q9.1: IM 通道优先级？哪些通道作为 P0 首批支持？
- Q9.2: 自主级别默认值？建议 Supervised 作为默认。
- Q9.3: IM 消息中的凭证（webhook token 等）如何安全存储？

### 第 2 部分: 技术栈与工具选型 (30 分钟)

**Speaker**: DevOps & 技术主管

#### 话题 4: 本地推理引擎 (8 分钟)

**候选方案**:
1. Ollama (推荐)
   - ✅ 简单易用, Docker 运行
   - ✅ 支持 GPU 加速
   - ✅ 模型生态丰富
   - ❌ 不支持 Windows 好

2. LocalAI
   - ✅ 完整 OpenAI API 兼容
   - ✅ 支持更多模型格式
   - ❌ 文档欠缺, 社区小

3. ONNX Runtime
   - ✅ 高度优化
   - ✅ 跨平台支持
   - ❌ 需要手动模型转换

**决议**:
```
推荐: Ollama
备选: LocalAI (如果需要 API 完全兼容)
```

**讨论问题**:
- Q10: 是否所有服务器都有 GPU? 预期 VRAM?
  - Mistral 7B: 需要 8GB VRAM (fp16)
  - LLaMA 13B: 需要 16GB VRAM (fp16)
  - 量化版本可减至 4-8GB
- Q11: 需要多 GPU 支持 / 模型并行吗?

#### 话题 5: 向量数据库 (8 分钟)

**候选方案**:
1. Milvus (推荐)
   - ✅ 本地部署, 完全可控
   - ✅ 高性能 (单机 100M+ 向量)
   - ✅ 支持混合检索 (向量 + 标量)
   - ❌ 需要 Etcd + 可观测性复杂

2. Weaviate
   - ✅ REST API 友好
   - ✅ 内置混合检索
   - ❌ 单机性能不如 Milvus

3. FAISS (SQLite 方案)
   - ✅ 轻量级, 适合演示
   - ❌ 不支持分布式, 不适合生产

**决议**:
```
推荐: Milvus (生产环境)
备选: FAISS + SQLite (MVP 阶段)
```

**讨论问题**:
- Q12: 预期存储多少向量? 多大数据量?
- Q13: 需要实时索引更新吗?
- Q14: 向量优先级: 精度 vs 速度 (L2 vs Cosine)?

#### 话题 6: 云服务商选择 (7 分钟)

**候选方案**:
1. OpenAI (GPT-4)
   - ✅ 最强能力
   - ✅ API 成熟稳定
   - ❌ 成本高 (~$0.03/1K tokens)

2. Anthropic Claude
   - ✅ 强大的文本理解与安全
   - ✅ 长 context (100K)
   - ❌ 成本中等 (~$0.01/1K tokens)

3. 私有部署 (vLLM + 自有服务器)
   - ✅ 完全可控, 无成本
   - ✅ 适合离线环境
   - ❌ 需要额外 GPU 资源

**决议**:
```
推荐: OpenAI GPT-4 (如果有预算)
备选: Claude (成本优化)
可选: 私有部署 (完全离线)
```

**讨论问题**:
- Q15: 月度云推理预算上限?
- Q16: 优先考虑成本还是质量?
- Q17: 是否需要支持离线模式?

#### 话题 8: 技术风险 (10 分钟)

**风险点**:

1. **本地推理性能不足** (概率: 中)
   - 缓解: 提前 benchmark (目标 < 500ms P95)
   - 备选: 微调模型 / 量化

2. **向量检索召回率低** (概率: 中)
   - 缓解: 混合检索 (向量 + BM25 + LLM 重排)
   - 备选: 定期微调嵌入模型

3. **云 API 成本超预算** (概率: 高)
   - 缓解: 严格路由策略 + 配额限制
   - 备选: 自有模型部署

4. **向量 DB 容量瓶颈** (概率: 低)
   - 缓解: 定期分片 / 数据归档
   - 备选: 升级到 Qdrant (更好扩展性)

**讨论问题**:

## 常见问题与现场回答

### Q1: 为什么必须本地优先?

**A**: 隐私与成本。本地处理避免敏感数据的网络传输与云存储，同时节省 API 成本。云端用于补充，不是替代。

### Q2: 如何保证脱敏的完整性?

**A**: 三重防线:
1. 多种 PII 检测器 (正则 + ML)
2. 所有云调用前的强制检查
3. 审计日志记录所有脱敏操作

### Q3: 本地推理速度能达到用户期望吗?

**A**: 取决于硬件。基准:
- Mistral 7B (8GB GPU): ~100-200ms per token → 平均 2-5s 句子
- 可通过量化降至 4-bit 加速

需要提前 benchmark 你的硬件。

### Q4: 如果云 API 不可用怎么办?

**A**: 自动降级到本地或混合。系统设计始终有本地备份。

### Q5: 向量数据库多大才需要分片?

**A**: Milvus 单机能处理 100M+ 向量。只有超过这个规模或对延迟要求极高时才需要分片。
