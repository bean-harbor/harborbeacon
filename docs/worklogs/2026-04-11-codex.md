## 今日完成

### 1. 排查并修正 Clash / 系统代理对局域网发现的影响

已检查并确认：

- `ClashX Pro` 当前选中的订阅文件是 `~/.config/clash/accessflow.grx.rent.yaml`
- 同时发现界面上还有 `微斯人，吾谁与归.yaml` 这一份同源配置
- 系统代理绕过列表已包含 `192.168/10/172` 与 `*.local`

已处理：

- 为两份 Clash 配置都补上内网、组播、广播直连规则
- 为 DNS 增加 `fake-ip-filter`，避免 `*.local` 等本地域名被 fake-ip 接管
- 额外确认当前 macOS 各网络服务的系统代理绕过项配置

结论：

- 普通局域网地址访问未被 Clash 完全阻断
- 自动发现更可能受广播 / 组播 / 私有发现协议影响，而不是普通 TCP 访问

### 2. 直接验证 IPC 基础连通性

目标设备：

- `192.168.3.73`

本机网络状态：

- Mac 当前地址为 `192.168.3.53`
- 与 IPC 同处 `192.168.3.0/24`

实测结果：

- `ping 192.168.3.73` 正常
- `8000/tcp` 可连接
- `1900/udp` 与 `3702/udp` 可达
- `554/tcp` 明确拒绝连接
- 常见 HTTP / HTTPS / RTSP 端口默认未开放

结论：

- 摄像头在线
- 局域网链路正常
- 但标准 RTSP 服务未启用

### 3. 验证萤石本地 RTSP 与海康兼容控制面

已用本地验证码：

- `admin / MZBEHH`

尝试内容：

- 常见 RTSP 地址探测
- 海康 / 萤石 `ISAPI` 路径访问
- GitHub 开源项目 `ylemoigne/ezviz-enable-rtsp`
- 海康 SDK `servicesSwitch` 开关 RTSP
- 海康 SDK `RealPlay` 常规预览参数组合

结果：

- RTSP 始终因 `554` 未开启而失败
- 开源 `ezviz-enable-rtsp` 在 Linux 容器内可运行
- 但 `Set Service Switch failed : 17`
- 继续探测 `deviceInfo / capabilities / streaming` 等路径，仍统一返回错误 `17`
- 再尝试 `NET_DVR_RealPlay_V40` 常见 `channel / streamType / linkMode` 组合，也没有收到任何码流回调

关键结论：

- 该设备接受某种海康系登录
- 但当前固件/模式下未向我们开放标准 RTSP / ISAPI / 常规海康预览链路

### 4. 确认具体机型并重新评估接入策略

用户提供型号：

- `CS-C6c`

结合官方资料与实测结论：

- `CS-C6c` 更偏向萤石私有协议 / 云协议路径
- 暂未确认该机型在当前固件下可稳定开启本地 RTSP
- 第三方软件（Camlytics / iSpy / Agent DVR 等）大多仍依赖 RTSP / ONVIF

额外结论：

- `Camlytics` 虽列出 `Ezviz C6c`，但官方为 `Windows only`
- 不适合作为当前这台 Mac 上的直接验证路径

## 当前判断

- `CS-C6c` 不适合作为 HarborBeacon 第一批基于 `RTSP/ONVIF` 自动发现接入的样板机
- 若继续支持该机型，较可能需要：
  - 专门的 EZVIZ / Hikvision 私有协议桥接
  - 或借助官方客户端链路做更深入的协议逆向与行为验证

## 对 HarborBeacon 的建议

- 第一阶段应优先围绕开放型 IPC：
  - `ONVIF`
  - `RTSP`
  - 局域网标准发现协议
- 对 `CS-C6c` 这类萤石机型，可后续单独规划 `vendor_sdk` / `private_protocol` 路线
