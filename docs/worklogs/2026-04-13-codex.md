# 2026-04-13 Codex

## 今日目标

- 收口 PR review 中指出的管理面安全问题
- 确保飞书配置页和状态接口不会再泄露敏感信息
- 强化本地直连预览入口，避免被反向代理头部绕过

## 今日完成

### 1. `/api/state` 不再返回敏感字段

已对管理后台状态快照增加脱敏逻辑：

- `feishu_bot.app_secret` 不再出现在接口响应中
- `defaults.rtsp_password` 不再出现在接口响应中

这样桌面端 Web UI 仍可读取状态，但不会把真实密钥和摄像头密码直接暴露给任意访问者。

### 2. 手机配置页不再回显已保存的飞书密钥

已调整 `/setup/mobile`：

- `App Secret` 输入框默认留空
- 页面文案明确说明“出于安全原因不会回显历史 secret”
- 仍保留当前 `App ID` 与绑定状态显示，方便用户确认当前设备绑定的是哪一个 Bot

这避免了“二维码被他人扫到后直接看见已保存 secret”的风险。

### 3. 本地管理面与原始预览入口加上更严格的直连校验

已把以下接口统一纳入“仅本机/局域网直连访问”约束：

- `/api/state`
- `/setup/mobile`
- 绑定二维码与绑定刷新接口
- 飞书配置、默认参数、手动添加摄像头、局域网扫描等管理接口

同时补充了对以下头部的拦截：

- `Forwarded`
- `X-Forwarded-For`
- `X-Forwarded-Host`
- `X-Forwarded-Proto`
- `X-Real-Ip`

这样即便请求最终落到本机端口，也不能通过“带转发头的反向代理/隧道”伪装成本地访问。

### 4. 增补回归测试

已新增并通过测试：

- 转发头存在时，本地管理接口与本地摄像头预览接口都会拒绝访问
- 飞书 Bot 配置脱敏后不再包含 `app_secret`

已把项目口径从“旧主链路 vs Home Agent Hub MVP 并行”收束成统一表述：

- `Local Agent V2` 重新定位为长期平台主干
- `Home Agent Hub` 明确为第一个正式垂直域
- `HarborBeacon` 明确为未来统一北向入口

本次新增/更新文档包括：

- `docs/platform-home-agent-hub-unified-architecture.md`

这样后续就不再是“摄像头产品自己旁路演进”，而是“平台主干承接 camera domain”。

### 6. 补上最小 Assistant Task API，打通 HarborBeacon -> camera 域

已新增本地 Task API 与最小 camera domain 主链路：

- 新增 Rust 二进制：`assistant-task-api`
- 新增运行时模块：
  - `src/runtime/task_api.rs`
  - `src/runtime/task_session.rs`
- HarborBeacon 侧新增：
  - `harborbeacon/task_api.py`
  - `harborbeacon/camera_domain.py`
  - `skills/builtins/home.camera_hub/skill.yaml`

当前已打通三条最小链路：

- `camera.scan`

- `camera.analyze`

其中：

- `camera.scan` 复用现有 `CameraHubService::scan`
- `camera.connect` 复用现有 `CameraHubService::manual_add`
- `camera.analyze` 复用现有 `VisionExecutor`

### 7. 把旧飞书 Bot 的关键多轮接入逻辑下沉到平台接入层

这次没有重写旧 `feishu-harbor-bot`，而是先把最关键、最有复用价值的一段逻辑下沉：

- `扫描摄像头`
- `接入 1`
- `密码 xxxxxx`

现在这套多轮流程已经可以由 HarborBeacon 会话层 + Task API 会话存储承接，不再只能依赖旧 bot 内部状态。

实现效果：

- HarborBeacon regex parser 已可识别 `扫描摄像头 / 接入 1 / 分析客厅摄像头`
- dispatcher 已可保存 `resume_token`
- 当 Task API 返回 `needs_input` 时，会提示用户回复密码
- 用户回复 `密码 xxxxxx` 后会续执行 `camera.connect`

### 8. 调整路由与执行器匹配规则，避免跨域串 executor

为了让 `service.*` 和 `camera.*` 能共存在 HarborBeacon 路由里，本次还补了一个关键修正：

- Python 侧 router 现在会优先检查 executor 是否 `supports(action)`
- executor factory 现在可以按 capability 约束执行器适用范围

这样：

- `service.*` 不会误落到 camera Task API executor
- `camera.*` 也不会被现有 HarborOS service executor 错吃

### 9. 增补 HarborBeacon / Task API 定向测试

已新增并通过测试：

- HarborBeacon camera intent 识别
- HarborBeacon dispatcher 的密码补参续执行
- Task API Python client payload 构造
- router 的 capability-aware executor 选择
- Rust 侧 Task API / task session 单测
